Meet Dejavu

Dejavu is an online fashion store. It's a full-stack web application where customers browse clothing, add items to their cart, and check out with a credit card.

Think of it like any online store you've used — but simpler. No recommendations engine, no social features, just the core: show products → let people buy them → collect payment.

In this course, you'll trace exactly what happens when someone clicks "Add to Cart" on a jacket. You'll see how five different parts of the system talk to each other to turn that click into a confirmed order and a charge on their credit card.

Simple, right? But under the hood, five different components are coordinating to make this happen. Each one has a specific job, and they talk to each other in a very specific order.

What's Really Happening Behind the Scenes

When you click "Add to Cart," here's the chain reaction that follows:

🖥️

Your Browser Reacts

The item gets added to a list stored in your browser's memory. No internet request yet!

📦

Server Checks Stock

When you click "Checkout," the backend verifies there's enough inventory before charging you.

🗄️

Database Confirms Stock

The server asks the database: "How many Medium jackets do we have left?" Database responds with the count.

💳

Stripe Takes Payment

You're redirected to Stripe's secure page to enter your credit card. Dejavu never sees your card number.

🔔

Webhook Confirms Order

After payment succeeds, Stripe calls back to the server: "Hey, they paid!" The server creates an order and decrements stock.

💡
Why This Matters for You

Understanding this flow helps you debug problems faster. If a user says "I clicked checkout but nothing happened," you now know to check: Did the frontend send the request? Did the backend receive it? Did stock validation pass? Did Stripe respond? Each step has a specific place in the chain.

Meet the Cast

The five components that make Dejavu work, and what each one does

The Five Main Characters

Every e-commerce store needs these five roles. Think of them as a team where each member has a specific job:

F

Frontend

The UI you see and click. Built with React. Runs in your browser.

B

Backend

The business logic server. Built with Node.js + Express. Validates requests, checks stock, creates checkout sessions.

D

Database

The long-term memory. Uses Supabase (Postgres). Stores products, variants, stock, orders, users.

S

Stripe

The payment processor. Handles credit cards securely. You never touch card numbers — Stripe does all the heavy lifting.

W

Webhook

The callback listener. When payment completes, Stripe calls your server with the news. Then you create the order.

💡
Separation of Concerns

This pattern — splitting responsibilities into focused roles — is one of the most important ideas in software engineering. Engineers call it "separation of concerns." Each component does one thing well, which makes the system easier to debug, test, and change. If the payment processor breaks, you know exactly where to look.

03

The Product Catalog

How products, sizes, and inventory live in the database

Two Tables Work Together

The store doesn't just have "products" — it has products with sizes. A single jacket might come in Small, Medium, and Large. Each size has its own stock count.

To handle this, the database uses two tables:

Product Table

id Unique ID
name "Cargo Jacket"
price 99.50
images Array of URLs
description JSON text

The general info: what it is, what it costs, what it looks like.

ProductVariant Table

id Unique ID
productId Links to Product
size "Medium"
stock 7
stripeProductId For payment

The specific details: which size, how many left, payment ID.

Think of it like a family tree: one Product (the parent) has multiple ProductVariants (the children). The Cargo Jacket is one product, but it has three variants: Small, Medium, Large.

💡
SKU is Just a Fancy Word for Variant

In retail, people call these SKUs (Stock Keeping Units). It's the same concept: a Product is the general item ("Cargo Jacket"), and each SKU/Variant is a specific sellable version ("Cargo Jacket, Medium"). When you tell AI to "add SKU tracking," you now know what that means!

How the Code Fetches Product Details

When you click on a product, the frontend asks the backend: "Give me all info for product X, including all available sizes." Here's the exact code that runs:

CODE 
let query = supabase
  .from("Product")
  .select("*, ProductVariant (*)")
  .eq("id", productId)
  .single();

const { data, error } = await query;
PLAIN ENGLISH

Start building a database query...

Look in the "Product" table...

Give me everything (*) from Product, plus all related rows from ProductVariant...

But only for the product where the ID matches this specific one...

And I'm expecting exactly one result (not an array).

Now actually send that query and wait for the answer.

The key line is "*, ProductVariant (*)" — that's a JOIN in disguise. It says "give me the Product data, and also grab all the ProductVariant rows that belong to it."

The response looks like this:

{
  id: "abc-123",
  name: "Cargo Jacket",
  price: 99.50,
  ProductVariant: [
    { id: "v1", size: "Small", stock: 3 },
    { id: "v2", size: "Medium", stock: 7 },
    { id: "v3", size: "Large", stock: 0 }
  ]
}

See how the ProductVariant field contains an array of all the sizes? That's the JOIN at work — one query, all the data you need.

The Shopping Cart Magic

Why your cart lives in your browser, not the database

The Cart Lives in Your Browser's Memory

When you click "Add to Cart," the item doesn't immediately go to the database. Instead, it's stored in React state — basically, a variable in your browser's memory.

Here's why:

⚡

Instant Feedback

No waiting for a server round-trip. Click "Add" and the cart updates immediately.

💾

No Database Clutter

People add/remove items constantly. If every cart change hit the database, you'd store thousands of abandoned carts.

🔒

Privacy

Your cart is yours until you checkout. No tracking, no "items in cart" analytics.

💡
The Tradeoff

Close the browser tab and your cart is gone. That's the cost of keeping it local. Big e-commerce sites (Amazon, Target) store carts in the database so they persist across devices — but they also have the infrastructure to handle millions of cart operations per second. For a small store, local state is simpler and faster.

The Add-to-Cart Code

Here's the function that runs when you click "Add to Cart":

CODE 
const handleAddToCart = (item) => {
  setCartItems((prev) => {
    const existing = prev.find(
      (i) => i.id === item.id
    );

    if (existing) {
      return prev.map((i) =>
        i.id === item.id
          ? { ...i, quantity: i.quantity + 1 }
          : i
      );
    }

    return [...prev, { ...item, quantity: 1 }];
  });
};
PLAIN ENGLISH

This function runs when the button is clicked...

Update the cart by looking at the current cart items...

Check if this item is already in the cart...

If yes, bump the quantity by 1...

If no, add it as a new item with quantity 1.

Notice it never calls fetch() to talk to the server. It just updates local state. The database only gets involved when you click "Checkout."

05

Taking Their Money

How Stripe handles payment processing securely

Why You Never Touch Credit Card Numbers

When the user clicks "Checkout," Dejavu doesn't ask for their credit card. Instead, it redirects them to Stripe — a separate company whose entire business is processing payments securely.

Here's why:

⚠️
Handling Credit Cards is a Legal Minefield

If you store credit card numbers, you must comply with PCI DSS — a massive set of security requirements. Audits, encryption, compliance officers, the works. For a small store, it's not worth it. Stripe is already PCI-compliant, so you let them handle it.

The flow looks like this:

User clicks Checkout
→
Backend creates Stripe session
Backend returns checkout URL
→
Frontend redirects user
User is now on Stripe's page
→
User enters card, clicks Pay
Stripe charges card
✓
Payment complete!

Dejavu never sees the card number. Stripe collects it, processes it, and tells Dejavu "payment succeeded" or "payment failed."

Creating a Checkout Session

Here's the backend code that creates the Stripe checkout session:

CODE 
const session = await stripe.checkout.sessions
  .create({
    mode: "payment",
    line_items: lineItems,
    success_url: `${FRONTEND_URL}/checkout/success`,
    cancel_url: `${FRONTEND_URL}/checkout/cancel`,
    shipping_address_collection: {
      allowed_countries: ["US", "CA"]
    }
  });

res.json({ checkoutUrl: session.url });
PLAIN ENGLISH

Ask Stripe to create a checkout session...

It's a one-time payment (not a subscription)...

Here are the items they're buying...

If payment succeeds, send them back to our success page...

If they cancel, send them to our cancel page...

Collect a shipping address (US or Canada only).

Send the checkout URL back to the frontend.

The session.url is a Stripe-hosted page where the user enters their card. The frontend gets that URL and redirects: window.location.href = checkoutUrl.

The Secret Handshake

Webhooks — the "reverse API" that confirms payment

The Problem: You Don't Know When Payment Completes

After redirecting the user to Stripe, your backend is out of the loop. The user enters their card on Stripe's page, not yours. When payment succeeds, Stripe knows — but you don't.

You could poll Stripe every few seconds: "Did they pay yet? Did they pay yet?" But that's wasteful and slow.

Instead, Stripe uses a webhook — it calls you back when payment completes:

Normal API
You call them: "Hey Stripe, give me product info"
Webhook
They call you: "Hey Dejavu, payment just completed!"

Think of it like push notifications. Instead of checking your email every minute (polling), your phone pings you when new mail arrives (webhook).

💡
Webhooks Are Everywhere

This isn't Stripe-specific — webhooks are a standard pattern for async events. GitHub uses them to notify you when someone opens a PR. Slack uses them to trigger bots. Shopify uses them when an order ships. Once you understand this pattern, you'll see it everywhere.

The Security Problem

Anyone can send an HTTP request to your webhook endpoint. What if someone fakes a "payment succeeded" message to get free stuff?

Stripe signs every webhook with a secret key. Your backend verifies the signature before trusting the message:

CODE 
const sig = req.headers["stripe-signature"];

try {
  event = stripe.webhooks.constructEvent(
    req.body,
    sig,
    WEBHOOK_SECRET
  );
} catch (err) {
  // Signature invalid - reject it
  return res.status(400).send("Bad signature");
}
PLAIN ENGLISH

Get the signature from the request header...

Try to verify this webhook came from Stripe...

Use the secret key only Stripe and you know...

If signature doesn't match...

Reject it immediately — someone's trying to fake a webhook.

The WEBHOOK_SECRET is like a password that proves the request really came from Stripe.

Creating the Order

Once the signature is verified, the webhook handler:

1

Creates an Order record

Stores the session ID, customer email, total amount, shipping address

2

Creates OrderItem records

One row per item purchased (jacket, variant ID, quantity, price)

3

Decrements stock

If they bought Medium, reduce the Medium variant's stock count by 1

Now the order is official. The database knows what was purchased, payment is confirmed, and inventory is updated. Done!

When Things Break

Common failure modes and how to debug them

Where Things Go Wrong

Every step in the purchase flow has potential failure points. Here are the most common ones:

Out of Stock

User adds size Medium to cart, but between adding and checking out, someone else bought the last one.

Where to look:

Backend checkout controller — stock validation happens there

Double Orders

Stripe webhook fires twice (network retry), creating two orders for the same payment.

The fix:

Idempotency check — before creating an order, check if one already exists for this session ID

Webhook Never Arrives

Payment succeeds, but your server is down or Stripe can't reach it. No order gets created.

The fix:

Stripe Dashboard logs all webhooks. Check there to see if delivery failed, then manually replay it

Network Timeout

Frontend sends checkout request, but backend takes too long. Request times out before Stripe session is created.

Where to look:

Backend logs — check if the request reached the server and how long the database query took
